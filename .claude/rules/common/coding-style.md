# Coding Style

## 일반 원칙
- 기존 코드 패턴과 컨벤션을 따른다
- 과도한 엔지니어링을 피한다 — 요청된 변경만 수행
- 불필요한 주석, docstring, 타입 어노테이션을 추가하지 않는다
- 한 번만 쓰이는 헬퍼/유틸리티/추상화를 만들지 않는다
- 비슷한 코드 3줄이 조기 추상화보다 낫다
- 사용하지 않는 코드는 주석 처리 대신 완전히 삭제

## 작업 범위와 판단
- 구현 전에 요청의 성공 기준과 검증 방법을 짧게 정한다
- 결과가 달라지는 불확실성은 `.claude/rules/common/clarification-protocol.md`에 따라 선택지와 trade-off를 제시하고 사용자에게 질문한다
- 결과에 영향이 작은 가정은 명시하고 진행한다
- 여러 해석이 가능하면 선택지와 트레이드오프를 드러낸다
- 더 단순한 접근이 충분하면 복잡한 설계를 만들지 않는다

## 변경 원칙
- 변경한 모든 라인은 사용자 요청과 직접 연결되어야 한다
- 요청받지 않은 인접 코드 정리, 포맷팅, 리팩토링을 하지 않는다
- 기존 스타일이 선호와 달라도 해당 코드베이스의 패턴을 따른다
- 내 변경으로 생긴 미사용 import, 변수, 함수만 제거한다
- 기존 dead code는 사용자가 요청하지 않는 한 삭제하지 않고 필요 시 언급만 한다

## 에러 처리
- 시스템 경계(사용자 입력, 외부 API)에서만 검증
- 발생할 수 없는 시나리오에 대한 에러 핸들링을 추가하지 않는다
- 내부 코드와 프레임워크 보장을 신뢰한다

## 보안
- OWASP Top 10 취약점 방지 (XSS, SQL Injection, Command Injection 등)
- API 키, 시크릿은 절대 커밋하지 않는다
- 보안 취약 코드를 발견하면 즉시 수정

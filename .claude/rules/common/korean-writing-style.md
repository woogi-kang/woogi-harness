# Korean Writing Style

한국어 기술 문서, 제출문, 지원서, 과제 답안, 고객 전달 문서를 만들거나 고칠 때 적용하는 표현 품질 규칙.

---

## 1. 적용 기준

다음 요청은 작성/수정 후 별도 한국어 표현 점검을 거친다.

- 한국어 문서, 제출문, 과제 답안, 지원서, 제안서, 회고, 보고서
- 영어 원문이나 AI 초안을 한국어로 옮긴 문서
- 기술/운영/데이터 용어가 많은 문서
- 사용자가 `번역투`, `직역`, `어색한 표현`, `현업 용어`, `용어 다듬기`를 언급한 경우

## 2. 기본 원칙

- 코드 식별자, JSON 필드, 상태값, CLI 명령, 파일명은 원형을 보존한다.
  예: `assigned_leaf`, `taxonomy_gap`, `schema_version`, `human_review_required`
- 설명문과 표의 prose는 독자가 실제 현업에서 쓰는 한국어로 풀어 쓴다.
- 영어 전문 용어가 필요한 경우 첫 등장에 쉬운 한국어 설명을 붙이고, 이후에는 일관된 표현을 쓴다.
- 기계 번역처럼 보이는 동사형 직역을 피한다. 특히 `잠근다`, `잠금`, `캡을 씌운다`, `게이트한다` 같은 표현은 문맥에 맞게 다시 쓴다.
- 한국어 조사와 호응을 최종 검토한다. 대량 치환 후에는 `기준를`, `요약와`, `대상로` 같은 조사 오류가 남기 쉽다.

## 3. 자주 고칠 표현

| 피할 표현 | 권장 표현 |
|---|---|
| 비용 잠금, 비용을 잠근다 | 파일럿 기반 예산 확정, 예산 상한 확정 |
| budget envelope | 예산 범위, 승인 예산 범위 |
| cost cap, cap | 비용 상한, 처리 한도 |
| gate | 통과 기준, 승인 기준, 중단 기준 |
| offline gate | 오프라인 통과 기준 |
| shadow inference | 섀도 추론, 병렬 추론 |
| canary | 카나리 배포, 소수 트래픽 배포 |
| rollback | 롤백, 이전 버전 복구 |
| leaf | 최하위 카테고리 |
| taxonomy | 분류 체계 |
| 저확신 | 확신도가 낮은, 낮은 확신도 |
| 사람 검수 | 인력 검수, 수동 검수 |
| 제한 서빙 | 제한 노출, 제한 제공 |
| unsupported rate | 근거 없는 응답 비율, 미지원 응답 비율 |
| fallback | 대체 경로, 대체 처리 |
| calibration | 확신도 보정 |
| recall | 재현율, 후보 포함률 |
| precision | 정밀도 |
| idempotency | 재실행 안전성, 중복 실행 방지 |
| checkpoint | 체크포인트, 재개 지점 |
| batch quota | 배치 처리 한도 |
| adjudication | 판정 조정 |
| 값 채움률 | 값 보유율, 추출 가능률 |

## 4. 점검 절차

1. 문서 수정 전 코드 블록, JSON, 백틱 식별자를 확인한다.
2. prose와 표를 먼저 다듬고, 코드 식별자는 원형으로 복원한다.
3. 아래 검색어로 번역투 후보를 훑고, 코드 식별자와 설명문을 구분해 판단한다.

```bash
rg -n "잠금|잠근|budget envelope|\\bgate\\b|threshold|저확신|사람 검수|제한 서빙|adjudication|채움|unsupported|fallback|cap|환류|멱등|idempotency|calibration|recall|precision|long-tail|shadow|canary|traffic|rollback" <target>
```

4. 대량 치환 후에는 조사 오류도 별도로 검색한다.

```bash
rg -n "기준를|요약와|요인와|정의과|대상로|확신도을|비율는|셋와|단위을|병합가" <target>
```

5. 최종 산출물에는 `멱등성`처럼 독자에게 낯선 용어를 단독으로 두지 않는다. reader-facing 문서에서는 `재실행 안전성`을 우선 쓰고, 필요하면 `재실행 안전성(idempotency)`처럼 병기한다.

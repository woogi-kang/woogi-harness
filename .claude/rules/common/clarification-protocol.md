# Clarification Protocol

불명확한 요청, 결과가 달라지는 선택지, 실행 전 승인 지점은 구조화된 선택 질문으로 정리한다. 목표는 사용자가 여러 안을 빠르게 비교하고 직접 고르면서 최적의 답에 도달하게 하는 것이다.

## 적용 시점

아래 중 하나라도 참이면 실행 전에 clarification round를 먼저 한다.

- 지시어의 대상이 불명확하다: "이거", "그거", "전에 하던 것", "정리해줘", "개선해줘"
- 실행 범위가 여러 해석을 가진다: 파일, 화면, 브랜치, 배포 환경, 문서 범위, 깊이
- 선택에 따라 결과물이 달라진다: 빠른 패치 vs 구조 개선, 단일안 vs 다중안, 로컬 검증 vs 배포까지
- 승인 전에는 위험한 작업이다: destructive git, production, 외부 전송, 비용 발생, 개인정보/시크릿 노출 가능성
- 에이전트/스킬/워크플로우가 사용자 선호를 알아야 한다

## 생략 기준

아래는 질문하지 않고 진행한다.

- 사용자가 파일, 명령, 대상, 범위를 명시했다.
- 단일 오타/포맷/명백한 버그 수정이다.
- 같은 세션에서 이미 확정한 작업의 계속 진행이다.
- 질문 비용이 작업 비용보다 크고, 결과 차이가 작다. 이때는 가정을 한 줄로 명시하고 진행한다.

## 질문 형식

런타임에 native 선택 UI가 있으면 그것을 우선한다. 예: Claude Code `AskUserQuestion`, Codex Plan mode `request_user_input`. 선택 UI가 없으면 Markdown으로 같은 구조를 제시한다.

각 질문은 아래 형식을 따른다.

- 질문 수: 한 번에 1-3개만 묻는다.
- 옵션 수: 질문당 2-4개만 둔다.
- 첫 옵션: 현재 정보 기준 추천안이며 라벨에 `(권장)`을 붙인다.
- 설명: 각 옵션은 즉시 결과, trade-off, 위험/비용을 한 문장으로 설명한다.
- 언어: 사용자 대화 언어를 따른다.
- 기타: 제공한 선택지 밖의 답을 허용한다. native UI가 `Other`를 제공하면 사용하고, Markdown이면 "직접 지정" 옵션을 둔다.

좋은 선택 질문 예:

```text
어떤 깊이로 진행할까요?

1. 빠른 흡수 (권장)
   공통 규칙과 라우팅 문서만 수정합니다. 오늘부터 동작이 바뀌지만 자동 선택 기록은 수동 캡처입니다.
2. 규칙 + 캡처까지
   GBrain 캡처 절차까지 연결합니다. 변경 파일이 늘고 검증 범위가 커집니다.
3. 전체 런타임 훅 설계
   Claude Code 훅 수준까지 설계합니다. 가장 완전하지만 별도 구현/테스트가 필요합니다.
4. 직접 지정
   원하는 범위나 제약을 직접 입력합니다.
```

## 추천안 배치

추천은 사용자의 목표, 현재 repo 상태, 과거에 확인된 선호, 위험도를 근거로 둔다. 시스템이 편한 방향을 추천하지 않는다.

- 과거 선택이나 memory를 근거로 추천하면 그 근거가 memory-derived임을 짧게 밝힌다.
- cold start라면 "현재 정보 기준" 또는 "기본값 기준"이라고 쓴다.
- 추천안 설명에는 추천이 성립하는 조건을 포함한다.
- 나머지 선택지를 의도적으로 나쁘게 쓰지 않는다.

## 진행 흐름

1. 모호성/선택지/승인 지점을 감지한다.
2. 필요한 경우 관련 파일, 브랜치, memory, 공식 문서를 먼저 조회한다.
3. 2-4개 선택지를 만든다.
4. 사용자가 선택하면 그 선택을 실행 계약의 Scope, Success criteria, Approval boundary에 반영한다.
5. 아직 100% 명확하지 않으면 더 좁은 follow-up 질문을 한 번 더 한다.
6. 충분히 명확하면 실행하고, 완료 보고에 어떤 선택을 기준으로 진행했는지 남긴다.

## 에이전트와 스킬 경계

하위 에이전트나 스킬은 사용자를 직접 붙잡고 질문하지 않는다. 필요한 입력이 빠졌으면 structured blocker를 반환한다.

```markdown
## Missing Inputs

| 항목 | 필요한 값 | 이유 |
|------|-----------|------|
| 대상 브랜치 | main / feature branch / 직접 지정 | merge 또는 diff 기준이 달라짐 |

**Blocker**: 위 입력 없이는 결과가 달라집니다. 오케스트레이터가 사용자에게 선택지를 제시해야 합니다.
```

오케스트레이터는 blocker를 받아 선택 질문으로 변환하고, 답변을 하위 에이전트/스킬 prompt에 주입해 다시 실행한다.

## Decision Memory

구조화 선택이 다음 작업에도 재사용될 선호라면 작업 후 GBrain capture 후보로 남긴다. 단, 일회성 상황 판단은 캡처하지 않는다.

캡처 후보:

- 반복되는 실행 깊이 선호: "빠른 패치보다 리뷰 포함", "커밋/푸시까지 완료"
- 도메인별 기본값: "한국어 문서는 자연스러운 실무 표현 우선"
- approval boundary: "배포 전 항상 prod smoke 확인"
- 하네스 라우팅 결정: "외부 repo 분석은 selective absorption 중심"

캡처하지 않는 것:

- 이번 한 번만 적용되는 파일/브랜치 선택
- 민감 정보, 계정 정보, private URL
- 검증되지 않은 추측

GBrain에 캡처할 때는 `.claude/rules/common/memory-engine.md`의 캡처 기준과 secret scan 규칙을 따른다.

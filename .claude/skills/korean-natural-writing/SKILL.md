# Korean Natural Writing

한국어를 영어식 문장에 단어만 바꾼 결과가 아니라, 해당 독자와 장르에서 실제로 쓰일 법한 글로 만든다. 목표는 AI 탐지 우회가 아니라 의미를 더 정확히 전달하고 작성자의 목소리를 살리는 것이다.

## 역할 경계

- 새 글에서는 기존 도메인 스킬이 사실, 전략, 메시지와 자료 조사를 맡고 이 스킬이 최종 문장 품질을 맡는다.
- 초안이 없는 새 글은 제품 정보나 요청문을 곧바로 문장 재료로 쓰지 않는다. 필수 주장·고정 요소, 선택 가능한 사실, 금지 주장·표현, 필수 CTA를 구분한 `source brief`를 먼저 만든 뒤 그 범위 안에서 작성한다.
- 기존 초안에서는 핵심 주장과 정보 구조를 먼저 보존한 뒤 필요한 범위만 다시 쓴다.
- 맞춤법만 확인할 때는 `korean-spell-check`, 기술 문서의 용어 직역만 고칠 때는 `korean-writing-style.md`를 우선한다.
- 마케팅 전략, 사실 검증, 법률 판단을 이 스킬의 문체 판단으로 대체하지 않는다.

## 먼저 읽을 자료

- 장르를 정한 뒤 [genre-contracts.md](references/genre-contracts.md)에서 해당 계약만 읽는다.
- AI 티나 번역투를 진단할 때 [slop-signatures.md](references/slop-signatures.md)를 읽는다. 표현 목록을 금칙어로 쓰지 않는다.
- 깊은 윤문 또는 후보 비교를 할 때 [evaluation-rubric.md](references/evaluation-rubric.md)를 읽는다.
- 프로젝트 문체 프로필을 만들 때 [korean-writing-profile.template.md](assets/korean-writing-profile.template.md)를 `.claude/korean-writing-profile.md`로 복사해 채운다.
- 평가표나 말뭉치 연동을 바꿀 때만 [research-basis.md](references/research-basis.md)를 읽는다.

## 문체 기준 우선순위

아래 순서로 가장 구체적인 근거를 우선한다.

1. 현재 요청에 포함된 좋은 예문, 싫은 예문, 금지 표현
2. 프로젝트의 `.claude/korean-writing-profile.md`
3. 제품·브랜드 문맥 파일과 실제 고객 언어
4. 이 스킬의 장르 계약

문체 프로필이나 좋은 예문이 없으면 장르 기본값을 사용할 수 있지만 이를 작성자 고유의 목소리라고 부르지 않는다. 공개·제출용 글에서 목소리 차이가 결과를 크게 바꾸면 예문을 받거나, 예문 없이 진행할 경우 내부 계약에 `voice_basis: genre_default`와 `human_review_required: true`를 남긴다.

유명 작가나 살아 있는 특정 인물의 문체를 복제하지 않는다. 사용자가 소유하거나 사용 허락을 받은 예문에서 관찰 가능한 특성만 추출한다.

## 처리 강도 결정

### 기본 윤문

짧고 위험이 낮은 문장에 사용한다.

1. 의미와 고정 요소를 잠근다.
2. 문제 구간을 진단한다.
3. 한 번 다시 쓴다.
4. 원문과 대조해 정보 손실을 확인한다.

### 깊은 윤문

사용자가 AI 티·번역투 제거, 깊은 윤문, 후보 비교, 충분한 토큰 사용을 직접 요구했거나 공개·제출·캠페인용 원고처럼 품질 비용이 큰 경우 사용한다. 일반적인 짧은 윤문 요청만으로 이 경로를 강제하지 않는다.

깊은 윤문은 더 많이 고치는 절차가 아니라 독립 초안과 의미 검증에 더 많은 계산을 쓰는 절차다. 최소 수정안 하나를 정답처럼 붙들거나, 화려한 재작성안을 당연한 개선으로 보지 않는다.

1. 의미 단위별 수정 예산을 정한다.
2. 서로 다른 두 편집 관점에서 동등한 지위의 초안을 독립적으로 만든다.
3. 각 변경의 의미 차이를 기록하고 후보를 익명으로 평가한다.
4. 순서 편향을 통제해 개선이 입증된 후보 또는 패치만 채택한다.
5. 최대 두 번만 고쳐 쓴다.

모든 한국어 한 줄에 깊은 윤문을 적용해 지연과 토큰을 낭비하지 않는다.

## 워크플로

### 1. 편집 계약 잠그기

초안을 고치기 전에 내부 작업 메모로 다음을 적는다.

- 글의 목적, 장르, 매체, 독자, 독자가 취할 행동
- 화자와 독자의 관계, 높임 수준, 종결 문체
- 분량과 형식 제약
- 바꾸면 안 되는 주장, 고유명사, 수치, 날짜, 링크, 인용, 코드 식별자
- 문체 프로필에서 가져온 핵심 특성 3개 이내

원고 상태에 따라 의미 원장을 다르게 만든다.

- `revision`: 원문을 의미 원장으로 삼고 핵심 주장, 조건, 부정, 책임 주체와 고정 요소를 잠근다.
- `new_draft`: 도메인 스킬이나 사용자가 제공한 자료에서 `source brief`를 만든다. brief는 `must_include_claims`, `must_include_fixed`, `allowed_facts`, `forbidden_claims`, `forbidden_tokens`, CTA와 독자 행동으로 나눈다. 필수 주장과 고정 요소, 선택 가능한 재료, 쓰면 안 되는 내용을 한 목록에 섞지 않는다. 출처가 없는 수치·성과·감정·장면은 만들지 않는다.

새 글의 두 편집본도 같은 `source brief`를 사용한다. 후보의 각 핵심 주장은 brief 항목으로 역추적할 수 있어야 하며, brief 밖의 주장은 문체가 좋아도 탈락시킨다.

깊은 윤문에서는 중요한 의미 단위와 문제 구간에 수정 예산을 붙인다.

- `preserve`: 이미 역할을 다하므로 그대로 둔다.
- `local_patch`: 특정 어휘, 호응, 어순, 호흡만 고친다.
- `restructure`: 국소 수정으로 목적이나 논리를 회복할 수 없을 때만 정보 순서나 문장 수를 바꾼다.

`restructure`는 자연스럽게 보인다는 이유만으로 선택하지 않는다. 장문에서는 모든 문장에 표를 만들지 말고 핵심 주장과 고위험 문단에만 적용한다.

결과를 크게 바꾸는 값이 비어 있거나 불명확하면 사용자에게 장르나 톤을 짧게 확인한다. 사소한 값은 장르 계약의 기본값을 쓰고 가정을 밝힌다.

### 2. 고치기 전에 진단하기

문장별 오탈자보다 먼저 글의 흐름을 본다.

- 한국어 독자가 기대하는 순서로 정보가 전개되는가?
- 문장마다 화자, 대상, 행동이 잡히는가?
- 추상 명사와 과장된 평가가 구체적 장면이나 근거를 대신하는가?
- 같은 길이, 같은 종결어미, 같은 삼단 구조가 반복되는가?
- 영어식 피동, 대명사, 전치사 대응 표현이 문맥 없이 남았는가?
- 카피 공식이나 감정 표현이 내용보다 앞서는가?

긴 글이나 반복 진단에는 다음 스크립트를 사용한다.

```bash
python3 .claude/skills/korean-natural-writing/scripts/analyze_korean_style.py <draft>
```

스크립트 결과는 수정 후보를 찾는 신호다. 플래그 수를 자연스러움 점수로 취급하지 않는다.

### 3. 두 편집 관점으로 독립 초안 만들기

단어 치환만 하지 말고 문단의 정보 순서, 주어, 동사, 호흡을 한국어 문맥에 맞게 다시 잡는다.

깊은 윤문에서는 다음 두 관점을 동등하게 사용한다.

- `장르 편집본`: 잠근 의미와 재료 안에서 정보 순서, 주어와 동사, 문장 호흡을 해당 장르의 실제 한국어에 맞게 다시 잡는다. 허용된 구간은 `restructure`할 수 있지만 새로운 주장, 감정, 장면, 카피 전략을 만들지 않는다.
- `보존 편집본`: 원문의 목소리와 정보 순서를 최대한 유지하고, 독자에게 실제 훼손을 주는 구간만 `local_patch`한다. 이미 역할을 다하는 문장은 그대로 둔다.

가능하면 두 편집본을 서로의 결과를 보지 않은 별도 에이전트나 분리된 문맥에서 만든다. 한 문맥에서 만들어야 한다면 각 관점의 작업 메모와 초안을 먼저 독립적으로 완성하고, 비교 전에는 한 편집본을 다른 편집본의 수정 기준으로 삼지 않는다. 어느 관점이 우위인지, 어떤 편집본이 스킬 적용본인지 후보에 표시하지 않는다.

진단 목록은 초안 전 문제를 찾고 초안 후 회귀를 확인하는 데 사용한다. 목록을 보며 문장마다 금칙어를 치환하지 않는다. 카피의 전략 변경이나 스토리의 확장은 사용자가 창작·재구성을 허용했을 때만 별도 후보로 만든다.

장문은 전체 문서의 화자·논지·종결 문체 지도를 먼저 만든 뒤 의미 단위로 편집하고, 마지막에 문서 전체를 한 번 읽는다. 모든 문단에 복수 후보를 만들지 말고 어색함이 크거나 역할이 중요한 문단에 집중한다.

### 4. 의미 보존 hard gate

후보를 평가하기 전에 원문과 대조한다.

```bash
python3 .claude/skills/korean-natural-writing/scripts/analyze_korean_style.py \
  <revised> --original <draft> --strict-invariants
```

종료 코드 `1`은 수치·링크·인용·코드 같은 고정 요소의 변형이므로 후보를 탈락시킨다. `2`는 부정 표현 수가 달라진 의미 위험 신호다. 형태만 달라진 동치 표현도 있으므로 자동 탈락시키지 말고 독립 비평자가 부정과 화행을 문장 단위로 확인할 때까지 선택을 보류한다.

다음 중 하나라도 생기면 문체 점수와 무관하게 탈락시킨다.

- 사실, 인과, 조건, 부정, 책임 주체의 변형
- 고유명사, 수치, 날짜, 링크, 인용, 코드 식별자의 누락·추가
- 근거 없는 성과, 통계, 추천사, 감정, 장면의 창작
- 요청한 형식, 분량, CTA의 훼손

자동 대조 전에 변경마다 `semantic_delta`를 내부 기록한다.

```text
[원문→수정] 실제로 바뀐 구간
[변화 축] scope / modality / negation / actor / temporal_scene / interpretation / discourse_mode / register
[허용 근거] 편집 계약이 허용한 변화 또는 진단된 훼손
[판정] licensed / unlicensed
```

`unlicensed`가 하나라도 있으면 후보를 버린다. 특히 다음을 동치로 취급하지 않는다.

- 기준 확인과 통과 여부 확인
- 가능 안내와 행동 요청, 권고와 의무
- 중립 행동과 의도를 해석한 행동
- 정지된 장면과 새로 만든 시간 흐름·감각·움직임
- 문어체와 축약형 구어체

자동 검사가 모든 의미 변화를 잡는다고 가정하지 않는다. 문장 단위 의미 대조를 함께 수행한다.
초안이 없는 새 글은 존재하지 않는 원문과 대조하지 않는다. 대신 `must_include_claims`를 후보의 실제 구간에 연결하고, 후보의 각 사실 주장을 `must_include_claims` 또는 `allowed_facts`로 역추적한다. `must_include_fixed`만 후보와 fixed-token gate로 대조한다. `allowed_facts`는 사용할 수 있는 재료이지 포함 의무가 아니다. `forbidden_tokens`가 하나라도 있거나 `forbidden_claims`·출처 없는 세부가 발견되면 탈락시킨다. brief 전체와 후보를 동일성 비교하지 않는다.
작가가 직접 쓴 대화를 윤문하는 등 인용부호 안의 문장 자체가 편집 대상이면 `--ignore-category quotation`을 명시한다. 외부 발언이나 출처 인용에는 사용하지 않는다.

### 5. 작성자와 비평자 분리하기

깊은 윤문에서는 실행 환경이 지원하면 별도 에이전트나 분리된 문맥의 비평자를 사용한다. 비평자에게는 원문, 편집 계약, 익명 후보, 평가표만 주고 작성 과정과 기대 정답은 주지 않는다. 별도 비평자를 쓸 수 있는데 자기검토만으로 대체하지 않는다.

1. 후보를 각각 독립적으로 점검한다.
2. hard gate를 통과한 후보만 A/B/Tie로 비교한다.
3. A/B 위치를 바꿔 한 번 더 비교한다.
4. 두 비교에서 같은 후보가 이기고 그 근거가 구체적이면 그 후보를 선택한다.
5. 두 비교가 다르거나 Tie이면 승자를 억지로 만들지 않는다. 두 편집본이 독립적으로 같은 방향으로 고친 패치, 또는 비평자가 목표 축의 개선을 구체적으로 입증한 패치만 남기고 논쟁적인 구간은 원문으로 되돌리거나 사람 검토 대상으로 둔다.

패치를 합친 결과는 새 후보로 취급한다. 의미 보존 hard gate를 다시 통과시킨 뒤, 어조와 정보 흐름이 한 글처럼 이어지는지 전체 문서를 한 번 소리 내어 읽는 속도로 점검한다. 문장 조각을 기계적으로 이어 붙이지 않는다.

별도 비평자를 사용할 수 없으면 새 문맥에서 같은 절차를 수행하고 `self_review_only`로 기록한다.

### 6. 최대 두 번만 고치기

비평은 구체적인 문제 구간, 훼손된 효과, 수정 의도를 제시해야 한다. 다음 중 하나면 멈춘다.

- hard gate를 통과하고 평가표의 합격 조건을 충족했다.
- 새롭고 국소적이며 실행 가능한 지적이 더 나오지 않는다.
- 직전 수정에서 순개선이 없거나 다른 평가 축이 나빠졌다.
- 두 번 수정했다.

반복할수록 더 좋아진다고 가정하지 않는다. 원문과 모든 중간본을 비교할 수 있게 유지한다.

## 결과 형식

기본 출력은 다음 Markdown 형식과 순서를 따른다.

1. `## 완성본` — 바로 사용할 수 있는 글
2. `## 다듬은 기준` — 글머리표 1~3개
3. `## 확인이 필요한 부분` — 사실이나 의도가 불명확할 때만 표시

내부 후보와 점수는 기본적으로 숨긴다. 사용자가 비교를 요청하면 익명 후보 2개와 차이만 보여준다.

## 품질 원칙

- 짧게 쓰는 것보다 정확한 호흡을 우선한다. 장르상 필요한 긴 문장, 피동, 명사화를 기계적으로 없애지 않는다.
- “자연스러운 한국어”를 하나의 중립 문체로 고정하지 않는다. 매체, 관계, 격식도, 목적에 따라 판단한다.
- 상투어 목록과 말뭉치 빈도를 정답표로 쓰지 않는다. 해당 문맥에서 표현이 맡는 일을 본다.
- 더 매끈하지만 누구의 글인지 알 수 없는 문장보다, 의미가 분명하고 목소리가 살아 있는 문장을 택한다.
- 원문이 이미 좋으면 불필요하게 바꾸지 않는다.
- slop 신호를 찾았다는 사실만으로 수정하지 않는다. 각 수정은 실제 독자 훼손 하나를 고쳐야 한다.
- 같은 어미의 짧은 문장 나열, 범용 CTA, 인위적 구체화, 이유 없는 축약형처럼 새 획일화를 만들면 해당 패치를 되돌린다.
- 토큰은 독립 초안, 계약, 의미 대조, 블라인드 후보 검증에 쓰며 수정량을 품질이나 노력의 증거로 삼지 않는다.
